# Git と GitHub のはじめかた（Windows版）

はじめての人向けに、**インストール → 最初のセーブ → GitHubにアップロード** までを順番にやってみよう。
コマンドは黒い画面（「ターミナル」と呼ぶよ）に打ち込む。むずかしそうに見えるけど、コピペでOK。

---

## 用語のミニおさらい

| 言葉 | かんたんに言うと |
|------|------------------|
| Git | 変更を何度もセーブできる道具（ゲームのセーブ機能） |
| GitHub | セーブデータをネット上に置く倉庫＋みんなで作業する場所 |
| リポジトリ（repo） | プロジェクト1つ分の入れ物フォルダ |
| コミット（commit） | 今の状態を1回セーブすること |
| プッシュ（push） | セーブをGitHubにアップロード |
| プル（pull） | GitHubの最新を自分のPCに取り込む |
| クローン（clone） | GitHubのリポジトリをまるごとコピーしてくる |

---

## STEP 1. Git をインストールする

1. ブラウザで **https://git-scm.com/download/win** を開く。
2. 自動でダウンロードが始まる（始まらなければ「64-bit Git for Windows Setup」をクリック）。
3. ダウンロードした `.exe` ファイルをダブルクリック。
4. インストール画面がたくさん出るけど、**基本ぜんぶ「Next」を押していけばOK**。最後に「Install」→「Finish」。

> 迷ったら設定は変えなくて大丈夫。初期設定のままで問題なし。

### インストールできたか確認

1. スタートメニューで「**Git Bash**」と検索して起動する（黒い画面が出る）。
2. 次を打って Enter：

```bash
git --version
```

3. `git version 2.xx.x` みたいに表示されたら成功！

---

## STEP 2. 名前とメールを登録する（最初の1回だけ）

コミット（セーブ）に「誰がやったか」を記録するための設定。Git Bash に打ち込む。

```bash
git config --global user.name "あなたの名前"
git config --global user.email "あなたのメール@example.com"
```

> 名前は英語でもローマ字でもOK。メールは GitHub に登録する予定のものにしておくと後がラク。

### 今設定されている名前・メールを確認する

```bash
git config --global user.name
git config --global user.email
```

それぞれ1行ずつ表示される。**何も出ずに次の行にいったら「まだ未設定」**（＝上の登録コマンドをやる必要あり）。まとめて全部見たいときは:

```bash
git config --global --list
```

`user.name=...` `user.email=...` のように一覧で出る。

### 名前・メールを上書き（変更）する

**同じコマンドをもう一度打つだけ**で上書きされる。`user.name` と `user.email` は値を1つしか持たないので、新しく設定すると前の値は自動で置き換わる。「削除してから登録」は不要。

```bash
git config --global user.name "新しい名前"
git config --global user.email "新しいメール@example.com"
```

上書きできたか、さっきの確認コマンドで新しい値が出ればOK。

### 補足：設定には2つの範囲がある

**`--global`（パソコン全体の初期設定）**
上のコマンドがこれ。特に指定しない限り、すべてのリポジトリでこの値が使われる。ふだんはこれでOK。

**リポジトリごとの設定（そのプロジェクトだけ変えたい場合）**
「仕事用のフォルダだけ会社のメールにしたい」みたいなとき。そのフォルダの中に入って、`--global` を外して打つ:

```bash
git config user.name "仕事用の名前"
git config user.email "仕事用@company.com"
```

そのフォルダの中だけ設定が上書きされ、他は `--global` のまま。**リポジトリごとの設定は `--global` より優先される**（そのフォルダの中ではリポジトリ設定が勝つ）。

今その場所で実際に使われる値を確認したいときは、`--global` を付けずに:

```bash
git config user.name
git config user.email
```

ふだん使いなら `--global` を上書きするだけで十分。

1. **https://github.com** を開く。
2. 「Sign up」からメール・パスワード・ユーザー名を登録。
3. 届いたメールで認証すれば完成。

## STEP 4. 練習用のフォルダを作ってセーブしてみる

ここからが本番。Git Bash で1行ずつ打っていこう。

### ① フォルダを作って移動する

```bash
cd ~/Desktop
mkdir git-practice
cd git-practice
```

- `cd` = 移動する（change directory）
- `mkdir` = フォルダを作る（make directory）
- これでデスクトップに `git-practice` フォルダができて、その中に入った状態。

### ② このフォルダを Git で管理し始める

```bash
git init
```

「このフォルダをリポジトリにするよ」という宣言。1回やればOK。

### ③ ためしにファイルを1つ作る

```bash
echo "はじめてのGit" > memo.txt
```

`memo.txt` という文字ファイルができる。

> 💡 **`>` は上書き、`>>` は追記**
> `>`（矢印1本）は「ファイルを空にして書き込む」＝毎回中身が丸ごと置き換わる。書き足したいときは `>>`（矢印2本）を使うと**末尾に付け足す**（既存の中身は消えない）。
> ```bash
> echo "7月6日 はじめてGitを触った" >> diary.txt
> echo "7月7日 pushができた" >> diary.txt
> cat diary.txt   # 中身を表示（2行になる）
> ```
> 日記や長い文章はエディタ（メモ帳など）で開いて書き、上書き保存してからコミットするのがラク。なお `>` / `>>` はGitではなくターミナル共通の書き方。

### ④ 今の状態を確認する

```bash
git status
```

変更されたファイルの一覧が出る。困ったら何度でも打っていい「今どうなってる？」ボタン。

### ⑤ セーブの準備（ステージングという）

```bash
git add memo.txt
```

「このファイルを次のセーブに含めるよ」と選ぶ操作。

- `git add memo.txt` … そのファイル**だけ**を含める。
- `git add .` … 今いるフォルダの中の**変更ぜんぶ**を含める。この「`.`（ドット）」は「今いるフォルダ（ここ）」を表す記号で、飾りではなく必要なもの。
- 注意：`add` とドットの**間に半角スペース**がいる。`git add .` は○、`git add.` は✕。

ふだんは全部まとめて `git add .` でOK。慣れてきて「この変更だけ先に保存したい」ときはファイル名で指定する。

### ⑥ セーブする（コミット）

```bash
git commit -m "最初のコミット"
```

`-m` のあとの「" "」の中がメモ（コミットメッセージ）。

> 💡 メッセージは「実際にやったこと」を自分の言葉で入れると、あとで `git log`（履歴）を見返したとき助かる。例：`"memoに2行目を追加"`、`"タイトルを修正"`。「更新」「2回目」のような中身の分からないメモは避けるのがコツ。

🎉 これで1回目のセーブ完了！

### 履歴（今までのコミット）を見る

```bash
git log
```

コミットのID・作成者・日時・メッセージが新しい順に並ぶ。画面が `:` で止まったら `q` キーで抜ける。1件を1行でスッキリ見たいときは:

```bash
git log --oneline
```

`e1c0cc0 最初のコミット` のように、短いID＋メッセージで一覧表示される。

## STEP 5. GitHub にアップロードする（push）

### ① GitHub 側に空の倉庫を作る

1. GitHub にログインして、右上の「＋」→「New repository」。
2. 「Repository name」に `git-practice` と入れる。
3. **公開範囲（Public / Private）を選ぶ** ← ここ大事！
   - **Public（公開）** … インターネットの誰でも中身を見られる。
   - **Private（非公開）** … 自分（と招待した人）だけ見られる。
   - 初期状態だと **Public** が選ばれていることが多い。パスワードや個人情報、人に見せたくないものを置くなら **Private** を選ぶ。練習用ならどちらでもOK。
4. 下の「Create repository」を押す。
   （READMEなどのチェックは付けなくてOK）

> ⚠️ **注意**：ファイル1つずつに公開設定があるわけではない。`memo.txt` のようなファイルが公開されるかどうかは、それが入っている**リポジトリ全体の Public / Private 設定**で決まる。何も気にせず進めると Public のまま世界中に公開されることがあるので、最初の選択に注意しよう。

### 公開範囲をあとから変える方法

すでに作ってしまっても、あとから切り替えられる。

1. GitHub でそのリポジトリのページを開く。
2. 上の「**Settings**」タブをクリック。
3. ページを一番下までスクロールして「**Danger Zone**」を探す。
4. 「**Change repository visibility**」→「Change visibility」を押す。
5. 「Make private（非公開にする）」または「Make public（公開にする）」を選ぶ。
6. 確認のためリポジトリ名の入力を求められるので、そのとおり打って確定。

### ② 作成後の画面に出るコマンドを使う

「…or push an existing repository」の欄に、こんな2行が表示される。**自分の画面に出たものをコピペ**するのが確実（ユーザー名が入るため）。だいたいこんな形：

```bash
git remote add origin https://github.com/あなたのユーザー名/git-practice.git
git branch -M main
git push -u origin main
```

- `remote add origin …` = 「この倉庫と自分のフォルダをつなぐ」
- `push` = アップロード実行

初回は GitHub のログインを求められることがある。画面の案内にしたがってサインインすればOK。

### ③ 確認

GitHub のページを再読み込みすると、`memo.txt` が表示されているはず。アップロード成功！

## STEP 6. これからの毎日の流れ

ファイルを編集したら、この3つをくり返すだけ：

```bash
git add .
git commit -m "何をしたか短く書く"
git push
```

他の人が更新した分を取り込みたいときは：

```bash
git pull
```

## 中断と再開（Git Bashを閉じる・PCを再起動する）

### そもそもGitは「常時監視」していない

フォルダが特別なのは、中に **`.git` という隠しフォルダ**があるから。これまでのコミット（履歴）は全部その中に**ファイルとして保存**されている。Gitは `git status` や `git commit` などの**コマンドを打った瞬間だけ**動き、その間は何も常駐していない。Git Bash は**ただのコマンド入力窓**にすぎない。

そのため:

- **Git Bash を閉じても** … リポジトリに影響なし。`.git` はディスクに残る。
- **PCをシャットダウンしても** … コミット済みの履歴もファイルも全部ディスクに保存されているので、何も失われない。次に開けば同じ状態から続けられる。

> 例え：`.git` はフォルダの中に置いてある「日記帳（保存データ）」。Gitはそれに書くときだけ手に取る道具、Git Bash は机。机から離れても日記帳は棚に残ったまま。

### 再開のしかた（この2ステップだけ）

1. スタートメニューから **Git Bash** を開く。
2. 作業フォルダに移動する:

```bash
cd ~/Desktop/git-practice
```

プロンプトの末尾に **`(main)`** と出ていれば、正しい場所（Git管理下のフォルダ）にいる印。あとは `git status` で状態を見て、いつもの `add → commit → push` を続けるだけ。

### 毎回やり直さなくていいこと

一度やった設定はディスクに保存されているので、再起動しても繰り返し不要:

- `git init`（`.git` はもうある）
- 名前・メールの設定（`--global` はPC全体に保存済み）
- `git remote add origin ...`（GitHubとのつなぎ先も保存済み）

つまり2回目以降は「Git Bashを開く → `cd` → 作業 → add/commit/push」だけ。

> 💡 コミットしていない変更はディスク上のファイルとしては残るが、「バージョン」としては記録されていない。区切りをつけたいときはシャットダウン前に `commit` しておくと安心。共同作業や別PCでも触るなら、作業開始前に `git pull` で最新を取り込む。

## STEP 7. 前のバージョンに戻す

Gitでの「1つ1つのバージョン」の正式名称は **コミット（commit）**。`commit` するたびに作られるセーブ1個が「1バージョン」で、それぞれに `96c1a98` のような **コミットID（ハッシュ）** が付く。「どこに戻すか」はこのIDで指定する。

戻し方はいくつかあるので、目的で選ぶ。初心者はまず**安全な方法（1・2）**を覚えればOK。

### 準備：戻したい先のIDを調べる

```bash
git log --oneline
```

```
96c1a98 2回目
e1c0cc0 最初のコミット
```

例えば「最初の状態に戻したい」なら目印は `e1c0cc0`。

### 方法1：ファイル1個だけ前の状態に戻す（いちばん安全・おすすめ）

`memo.txt` を「最初のコミットの中身」に戻す:

```bash
git restore --source=e1c0cc0 memo.txt
```

中身を確認して気に入ったら、いつもの流れで新しいコミットとして保存:

```bash
git add memo.txt
git commit -m "memoを最初の状態に戻した"
git push
```

履歴を消さないので、失敗しても安心。迷ったらこれ。

### 方法1.5：昔の中身を覗くだけ（戻さない）

```bash
git show e1c0cc0:memo.txt
```

ファイルは書き換わらず、画面に昔の中身が出るだけ。

### 方法2：あるコミットの変更を打ち消す（revert）

「あのコミットでやった変更をなかったことにしたい」とき:

```bash
git revert 96c1a98
```

履歴を消さず、**"打ち消し"の新しいコミットを追加する**やり方。push済みでも安全なので共同作業でも王道。実行するとメモ入力画面が出ることがある（そのままでよければ保存して閉じる）。あとで `git push`。

### ⚠️ 注意：`reset --hard` は当面使わない

検索で出てくる `git reset --hard ID` は「後のコミットをまるごと消して巻き戻す」強力な操作。**消したコミットは戻せず**、push済みだと厄介になる。慣れるまでは方法1・2で十分。

## 差分（変更点）の見方

「前のバージョンから何がどう変わったか」を色で見せてくれるのが **差分（diff）** 画面。GitHubのコミットをクリックしても見られるし、Git Bash でも `git diff` で見られる。テキストファイルをGitで管理する最大のメリットがこれ。

### 色分けの意味（ここが核心）

- **赤（ピンク）＋行頭の `−`** … **消された行**（前はあって、今は無い）
- **緑＋行頭の `+`** … **追加された行**（今回増えた）
- **色なし（白）** … 変わっていない行

行を書き換えた場合、Gitは「古い行を削除（赤）＋新しい行を追加（緑）」として表現する。

### 記号の読み方（例）

```
@@ -1 +1,2 @@
- 2026/07/06 00:46            ← 赤：この行が消えた
+ 2026/07/06 01:04 2nd        ← 緑：この行が増えた
+ 7月6日 はじめてGitを触った    ← 緑：この行も増えた
```

- **`@@ -1 +1,2 @@`** … 変更場所の見出し。`−` が変更前、`+` が変更後。「変更前は1行目から1行、変更後は1行目から2行」という意味。
- **`+2 -1`**（画面右上） … このファイルは **2行追加・1行削除** という要約。
- **左端の数字** … 行番号（赤側＝変更前、緑側＝変更後）。

> 💡 `>`（上書き）と `>>`（追記）の違いも差分に出る。上書きだと古い行が赤で消え、追記だと古い行は白のまま新しい行だけ緑で増える。
> なお `.docx` や `.slx` などバイナリ扱いのファイルは、この差分が見えない（巻末「ファイル形式とGitの相性」参照）。

## 公開する前に：個人情報のチェック

Publicリポジトリにpushすると中身は世界に公開される。**push前に「見られて困るものが入っていないか」を確認するクセ**をつけると安心。

### 確認したいもの

- 実名・メールアドレス・住所・電話番号
- パスワード、APIキー、トークンなどの秘密情報
- 会社や学校の内部情報

ファイルの中を文字で検索して確かめられる（例：自分の名前やメールを探す）:

```bash
grep -rn "探したい文字" .
```

何も出なければそのファイルにはその文字は入っていない、ということ。

### 気にしなくていいもの

- **コミットID（`96c1a98` など）** … コミットを見分けるただの識別番号。秘密情報ではなく、pushすれば履歴として公開されるものなので、資料などに書いてあっても問題ない。

### コミットの作者欄にメールを出したくないとき

Publicリポジトリでは、コミットの作者情報（`git log` のAuthor）に設定したメールが残る。本物のメールを見せたくなければ、GitHubの**匿名用メール**に変えておく手がある:

```bash
git config --global user.email "ユーザー名@users.noreply.github.com"
```

こうすると、これから作るコミットの作者欄に本物のメールが出なくなる。（GitHubの Settings → Emails で「Keep my email addresses private」を有効にすると、この専用アドレスが確認できる）

> ⚠️ うっかり秘密情報をコミット＆pushしてしまった場合、ファイルを消すだけでは**過去の履歴に残ったまま**になる。その場合は公開範囲をPrivateに変える・履歴から削除する等の対処が必要になるので、「push前チェック」で防ぐのがいちばんラク。

## こまったときメモ

- **打ち間違えた／よく分からなくなった** → `git status` で今の状態を確認。
- **コマンドが途中で止まって `:` みたいな画面になった** → キーボードで `q` を押すと抜けられることが多い。
- **貼り付け方** → Git Bash では右クリック、または `Shift + Insert` で貼り付け。
- **パスワードを聞かれ続ける** → GitHub は今はパスワードの代わりに「トークン」や「GitHub CLI」でのログインを推奨。うまくいかないときは「GitHub Desktop」というアプリを使うとボタン操作で push/pull できてラク。

### よく出るメッセージの意味（エラーではないもの）

- **`warning: ... LF will be replaced by CRLF ...`**（`git add` のとき）
  改行の書き方をWindows式（CRLF）にそろえるよ、というただの**お知らせ**。ファイルは壊れないし、そのまま次に進んでOK。毎回出るのが気になるなら一度だけ `git config --global core.autocrlf true` を打つと出なくなる。
- **`nothing to commit, working tree clean`**（`git commit` や `git status` のとき）
  「保存すべき変更が残っていない＝全部保存済み」という**良い状態**。もしファイルを編集したのにこれが出たら、エディタで**上書き保存（Ctrl+S）できていない**可能性。保存してから `git add .` をやり直す。
- **`Your branch is up to date with 'origin/main'.`**
  自分のPCとGitHubが同じ状態にそろっている、という意味。ズレなし＝理想的。
- **`(HEAD -> main, origin/main)`**（`git log` のとき）
  現在地・ローカルのmain・GitHubのmainが全部同じコミットを指している＝完全に同期できている印。
- **`remote origin already exists`**（`git remote add` のとき）
  originはもう登録済み、という意味。つなぎ先を変えたいなら `git remote set-url origin <URL>` を使う。

## もっとラクにやりたい人へ：GitHub Desktop を使う

コマンドが苦手なら、**GitHub Desktop** という無料アプリがおすすめ。
commit も push も pull も、文字を打たずにボタンでできる。ここまでのコマンド版と同じことを、画面のボタンでやるだけ。

### 1. アカウントを用意する

先に GitHub のアカウントが必要（STEP 3 と同じ）。まだなら https://github.com で「Sign up」から登録しておく。

### 2. インストールする

1. **https://desktop.github.com** を開く。
2. 「Download for Windows」をクリック。
3. ダウンロードした `GitHubDesktopSetup.exe` をダブルクリック。
4. インストールは自動で進む（特に何も選ばなくてOK）。少し待つとアプリが立ち上がる。

### 3. GitHub アカウントとつなぐ

1. 起動すると「Sign in to GitHub.com」というボタンが出る → クリック。
2. ブラウザが開くので、GitHub にログインして「Authorize（許可）」を押す。
3. 名前とメールの確認画面が出たら、そのまま「Finish」。

これでアプリとあなたの GitHub がつながった状態になる。

### 4. 最初のリポジトリを作ってみる

1. 画面の「**Create a New Repository on your hard drive**」を選ぶ。
2. 「Name」に好きな名前（例：`git-practice`）を入れる。
3. 「Local path」で保存する場所を選ぶ（デスクトップなどでOK）。
4. 「Create repository」を押す。

これで自分のパソコンにフォルダ（リポジトリ）ができる。

### 5. セーブ（commit）して GitHub にアップ（push）する

1. 作ったフォルダに、メモ帳などでファイルを1つ作って保存してみる。
2. GitHub Desktop に戻ると、左側に変更したファイルが自動で出てくる。
3. 左下の「Summary」欄に何をしたか短く書く（例：`最初のコミット`）。
4. 青い「**Commit to main**」ボタンを押す ← これがセーブ（commit）。
5. 上の方に出る「**Publish repository**」を押す ← これが GitHub へのアップロード（push）。

GitHub のページを見ると、ファイルが上がっているはず。成功！

### これからの流れ（GitHub Desktop版）

ファイルを編集したら、毎回この3ステップだけ：

```
Summaryを書く → 「Commit to main」 → 「Push origin」（右上のボタン）
```

コマンドを一切打たずに、Git の基本（commit・push・pull）が全部ボタンでできる。
仕組みを覚えたら、コマンド版とお好みで使い分けよう。

## おまけ：どんなファイルがGitに向いている？

Gitはどんなファイルでも「バージョンとして保存・復元」はできる。でも「**どこを変えたか（差分）を見る**」「**2人の変更を合体（マージ）する**」が得意なのは、中身が**テキスト**のファイルだけ。ここが向き不向きの分かれ目。

- **テキスト形式** … 差分もマージもできて相性◎。
- **バイナリ形式（＆ZIPで固めた形式）** … 保存・復元はできるが、Git単体では差分が読めずマージも不可（△）。

| ファイル | 中身 | 相性 |
|---------|------|------|
| プログラムのコード（.py .js .c など）、.m（MATLAB）、テキスト（.txt .md）、Gコード | テキスト | ◎ |
| STEP（.step / .stp） | テキストだが機械的 | ○（保存向き。差分は読みにくい） |
| Word/Excel/PowerPoint（.docx .xlsx .pptx） | XMLをZIP圧縮 | △ 中身はテキストでもGitにはバイナリに見える |
| Simulink（.slx）、ライブスクリプト（.mlx）、MATLABデータ（.mat）、図（.fig） | ZIP/バイナリ | △ |
| CADネイティブ（SOLIDWORKSの.sldprt など） | バイナリ | △ 大きくなりがち |

**△のファイルをうまく扱うコツ:**

- **Git LFS** … 大きなバイナリを別管理するGitの拡張。GitHubも対応（無料枠は小さめ）。
- **専用ツール** … CADなら **SOLIDWORKS PDM**、Simulinkなら **MATLABの比較・マージツール**（Comparison Tool / mlDiff・mlMerge）。Gitに設定すれば `.slx` でも差分・マージができる。
- **Office系** … 見た目の変更を追いたいなら、Word/PowerPointの「比較」機能や OneDrive/SharePoint のバージョン履歴のほうが得意。
- **MATLAB** … `.m` はテキストで相性◎。新しめのMATLAB（R2025a〜）はライブスクリプトを**テキストの.m形式**でも保存でき、Gitで差分が読める。

> ざっくり: **テキストほどGit向き、バイナリほど専用ツール向き**。迷ったら「メモ帳で開いて文字が読めるか」がひとつの目安。

## 用語おさらい（増補）

| 言葉 | 意味 |
|------|------|
| コミット（commit） | セーブ1個＝1バージョン。固有のID（ハッシュ）が付く |
| ブランチ（branch） | 本編を壊さず別ルートで作業する枝。最初は master か main |
| HEAD | 「今の現在地」を指す矢印 |
| origin | つなぎ先のGitHub倉庫の呼び名 |
| ステージング（git add） | 次のコミットに含めるものを選ぶ操作 |
| restore / revert | 前の状態に戻す（安全な戻し方） |

---

### まとめ（この順番だけ覚えればOK）

```
インストール → git init → git add → git commit → git push
```

あとは編集するたびに **add → commit → push** をくり返すだけ。おつかれさま！
